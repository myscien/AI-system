# 教师学情分析后端接口最小需求

本文档只覆盖教师学情分析页当前最需要支持的 2 个功能，供后端直接开始实现。

目标很明确：

1. 让老师快速判断班级整体学情
2. 让老师快速定位薄弱题目和需要关注的学生

不在本次范围内的内容：

- 知识点分析
- 历史趋势分析
- 学生个人详情页
- 复杂筛选、分页、导出

## 1. 本次只需要支持的 2 个功能

### 功能一：班级整体学情总览

教师进入分析页后，需要第一时间看到：

- 任务标题
- 应参与学生数
- 已提交学生数
- 已完成人数
- 完成率
- 班级平均正确率
- 低正确率学生数

这部分用于回答：

- 这次任务有多少学生参与了
- 班级整体完成情况怎么样
- 班级整体掌握情况怎么样

### 功能二：薄弱点定位

教师需要快速知道：

- 哪些题正确率最低
- 哪些学生正确率较低或未完成

这部分用于回答：

- 全班主要卡在哪几道题
- 哪些学生需要老师优先关注

## 2. 接口方案

为了减少前后端改动，本次建议仍然使用现有接口：

- `GET /api/tasks/{taskId}/analysis`

后端在当前接口基础上补充字段即可，前端先不新增接口。

## 3. 返回数据结构

建议接口返回 JSON 结构如下：

```json
{
  "task_id": 12,
  "task_title": "分数加减法练习",
  "assigned_students": 48,
  "submitted_students": 44,
  "completed_students": 39,
  "completion_rate": 0.8125,
  "average_accuracy": 0.71,
  "low_accuracy_student_count": 8,
  "question_stats": [
    {
      "question_id": "q1",
      "total_answers": 44,
      "correct_count": 20,
      "wrong_count": 24,
      "accuracy": 0.4545
    }
  ],
  "student_progress": [
    {
      "student_no": "20280101",

      "wrong_count": 24,

    }

```

## 4. 字段说明

### 顶部总览字段

- `task_id`
  - 任务 ID
- `task_title`
  - 任务标题
- `assigned_students`
  - 应参与学生总数
- `submitted_students`
  - 至少提交过 1 道题的学生数
- `completed_students`
  - 已完成整份任务的学生数
- `completion_rate`
  - 完成率，计算方式：`completed_students / assigned_students`
  - 返回 0 到 1 之间的小数
- `average_accuracy`
  - 班级平均正确率
  - 返回 0 到 1 之间的小数
- `low_accuracy_student_count`
  - 低正确率学生数
  - 低正确率判定阈值建议先固定为 `accuracy < 0.6`

### 题目统计字段 `question_stats`

每个元素至少包含：

- `question_id`
  - 题目 ID
- `total_answers`
  - 作答人数
- `correct_count`
  - 正确人数
- `wrong_count`
  - 错误人数
- `accuracy`
  - 该题正确率，返回 0 到 1 之间的小数

用途：

- 前端展示题目正确率表
- 前端生成薄弱题目列表

### 学生进度字段 `student_progress`

每个元素至少包含：

- `student_no`
  - 学号
- `student_name`
  - 学生姓名，没有可返回空字符串
- `answered_count`
  - 已作答题数
- `correct_count`
  - 正确题数
- `accuracy`
  - 当前正确率，返回 0 到 1 之间的小数
- `progress`
  - 当前完成进度，计算方式：`answered_count / 任务总题数`
  - 返回 0 到 1 之间的小数
- `status`
  - 学生状态，取值建议：
  - `not_started`
  - `in_progress`
  - `completed`
- `is_low_accuracy`
  - 是否属于低正确率学生
  - 布尔值

用途：

- 前端展示学生完成情况表
- 前端筛出低正确率学生和未完成学生

### 薄弱题目字段 `lowest_accuracy_questions`

每个元素结构与 `question_stats` 保持一致即可，建议后端直接返回按正确率升序排好的前 3 到 5 题。

用途：

- 前端直接展示“薄弱题目 Top N”
- 前端无需重复排序

## 5. 后端计算规则建议

### 学生状态判定

- `not_started`
  - 该学生 0 题作答
- `in_progress`
  - 已作答，但未完成全部题目
- `completed`
  - 已作答题数等于任务题目总数

### 低正确率学生判定

本次建议先用固定规则，避免前后端对口径理解不一致：

- `accuracy < 0.6` 视为低正确率学生

后续如果需要，可再改为可配置阈值。

### 薄弱题目判定

本次建议直接按题目 `accuracy` 从低到高排序，取前 3 到 5 题。

## 6. 最小实现要求

后端只要完成下面这些，就能支持前端先做出可用版本：

1. 补齐顶部总览字段
2. 在 `student_progress` 中补 `progress` 和 `status`
3. 保证 `question_stats` 中每题都有正确率
4. 返回 `lowest_accuracy_questions`
5. 所有比例字段统一返回 0 到 1 的小数，前端负责转百分比

## 7. 兼容性要求

为了减少前端适配成本，建议后端字段命名优先使用下划线风格：

- `task_id`
- `task_title`
- `assigned_students`
- `submitted_students`
- `completed_students`
- `completion_rate`
- `average_accuracy`
- `low_accuracy_student_count`
- `question_stats`
- `student_progress`
- `lowest_accuracy_questions`

如果后端已有驼峰字段，也可以继续保留，但建议同时返回下划线字段，方便前端统一处理。

## 8. 一句话结论

本次后端只需要围绕一个分析接口，补齐“班级整体总览”和“薄弱点定位”两类数据，就足够支撑教师学情分析页的第一版有效落地。
