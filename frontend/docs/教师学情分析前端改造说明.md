# 教师学情分析前端改造说明

本文档对应 [教师学情分析后端接口最小需求.md](C:\Users\wangyong\Desktop\ai_sys\frontend\docs\教师学情分析后端接口最小需求.md)，说明前端拿到这些新字段后，教师学情分析页应该如何消费。

本次只围绕 2 个功能改造：

1. 班级整体学情总览
2. 薄弱点定位

不在本次范围内：

- 知识点分析
- 趋势分析
- 复杂筛选
- 学生详情抽屉或弹窗

## 1. 相关文件

本次前端改造主要集中在：

- `src/api/tasks.js`
- `src/pages/TeacherAnalysisPage.vue`

如果需要补样式，再改：

- `src/style.css`

## 2. 当前页面现状

当前教师分析页已经有这些内容：

- 顶部任务信息
- 概览卡片
- 薄弱题目列表
- 题目正确率表
- 学生完成情况表

但当前展示还比较弱，主要问题是：

- 概览卡片没有体现完成率和班级平均正确率
- 学生表无法区分未开始、进行中、已完成
- 学生表无法直接看出哪些学生需要优先关注
- 参与人数目前还是基于旧字段 `total_students`

## 3. 前端需要消费的新字段

后端返回后，前端应优先消费下面这些字段：

- `task_title`
- `assigned_students`
- `submitted_students`
- `completed_students`
- `completion_rate`
- `average_accuracy`
- `low_accuracy_student_count`
- `question_stats`
- `student_progress`
- `lowest_accuracy_questions`

其中：

- 顶部总览区主要消费总览字段
- 薄弱题目区主要消费 `lowest_accuracy_questions`
- 学生表主要消费 `student_progress`

## 4. `src/api/tasks.js` 改造说明

### 4.1 扩展 `normalizeStudentProgressItem`

当前只标准化了：

- `student_no`
- `student_name`
- `answered_count`
- `correct_count`
- `accuracy`

需要补充：

- `progress`
- `status`
- `is_low_accuracy`

建议标准化后结构如下：

```js
function normalizeStudentProgressItem(item) {
  return {
    student_no: item.student_no ?? item.studentNo ?? "",
    student_name: item.student_name ?? item.studentName ?? "",
    answered_count: item.answered_count ?? item.answeredCount ?? 0,
    correct_count: item.correct_count ?? item.correctCount ?? 0,
    accuracy: item.accuracy ?? null,
    progress: item.progress ?? null,
    status: item.status ?? "",
    is_low_accuracy: item.is_low_accuracy ?? item.isLowAccuracy ?? false,
  };
}
```

### 4.2 扩展 `getTaskAnalysis`

当前只取了：

- `task_id`
- `task_title`
- `total_students`
- `question_stats`
- `student_progress`
- `lowest_accuracy_questions`

需要补充：

- `assigned_students`
- `submitted_students`
- `completed_students`
- `completion_rate`
- `average_accuracy`
- `low_accuracy_student_count`

建议最终返回结构至少包含：

```js
return {
  ...data,
  task_id: data.task_id ?? data.taskId ?? Number(taskId),
  task_title: data.task_title ?? data.taskTitle ?? data.title ?? "",
  assigned_students: data.assigned_students ?? data.assignedStudents ?? 0,
  submitted_students: data.submitted_students ?? data.submittedStudents ?? 0,
  completed_students: data.completed_students ?? data.completedStudents ?? 0,
  completion_rate: data.completion_rate ?? data.completionRate ?? null,
  average_accuracy: data.average_accuracy ?? data.averageAccuracy ?? null,
  low_accuracy_student_count:
    data.low_accuracy_student_count ?? data.lowAccuracyStudentCount ?? 0,
  question_stats: ...,
  student_progress: ...,
  lowest_accuracy_questions: ...
};
```

说明：

- 本次不建议继续依赖 `total_students` 作为主要统计口径
- 如果为了兼容旧数据，可以保留 `total_students`

## 5. `TeacherAnalysisPage.vue` 改造说明

## 5.1 顶部概览卡片改造

当前概览卡片是：

- 任务 ID
- 参与学生数
- 题目统计数
- 薄弱题目数

建议改成更符合老师判断逻辑的 4 张卡片：

- 应参与学生数：`assigned_students`
- 已提交学生数：`submitted_students`
- 完成率：`completion_rate`
- 班级平均正确率：`average_accuracy`

如果想保留第 5 张卡片，可增加：

- 低正确率学生数：`low_accuracy_student_count`

这样老师第一眼能判断：

- 有多少人没参与
- 有多少人没完成
- 班级整体掌握度如何

## 5.2 页面可增加两个计算属性

建议在页面里新增：

```js
const lowAccuracyStudents = computed(() => {
  return analysisStudentProgress.value.filter((item) => item.is_low_accuracy);
});

const incompleteStudents = computed(() => {
  return analysisStudentProgress.value.filter(
    (item) => item.status === "not_started" || item.status === "in_progress"
  );
});
```

用途：

- 一个用于展示“低正确率学生名单”
- 一个用于展示“未完成学生名单”

## 5.3 薄弱题目区的消费方式

当前页面已有 `weakQuestions` 计算属性，并且已经优先使用后端返回的 `lowest_accuracy_questions`。

本次建议保持这个策略不变：

- 如果后端返回了 `lowest_accuracy_questions`，直接使用
- 如果后端没返回，再前端从 `question_stats` 里兜底排序

这样前后端都更稳。

建议展示内容保持简单：

- 题目 ID
- 正确率

如果后续需要，再扩成卡片或表格。

## 5.4 学生表改造

当前学生表只展示：

- 学号
- 姓名
- 已答题数
- 正确题数
- 正确率

建议最少补 2 列：

- 完成进度：消费 `progress`
- 状态：消费 `status`

状态建议前端转成中文：

- `not_started` -> 未开始
- `in_progress` -> 进行中
- `completed` -> 已完成

完成进度建议直接显示百分比：

- `formatPercent(item.progress)`

这样老师可以快速区分：

- 没做
- 做了一半
- 做完但正确率低

## 5.5 增加“需要关注学生”区块

在“薄弱题目”下方或“学生完成情况表”上方，建议新增一个简短区块：

- 低正确率学生名单
- 未完成学生人数

展示方式不需要复杂，第一版可以直接用两个简单列表：

- 低正确率学生：姓名/学号 + 正确率
- 未完成学生：姓名/学号 + 状态 + 进度

这样页面从“统计页”变成“可行动页”。

## 5.6 `analysisHasData` 判定逻辑改造

当前逻辑主要依赖：

- `total_students`
- `question_stats`
- `student_progress`

建议改成优先看新字段：

```js
const analysisHasData = computed(() => {
  if (!analysisData.value) return false;
  return (
    Number(analysisData.value.assigned_students || 0) > 0 ||
    Number(analysisData.value.submitted_students || 0) > 0 ||
    analysisQuestionStats.value.length > 0 ||
    analysisStudentProgress.value.length > 0
  );
});
```

## 6. 前端展示优先级

本次建议前端按下面顺序落地，不要一次改太多：

1. 接口封装先兼容新字段
2. 顶部总览卡片改成真正有意义的 4 张卡
3. 学生表增加 `progress` 和 `status`
4. 新增“低正确率学生名单”或“需要关注学生”区块
5. 保持薄弱题目区继续消费 `lowest_accuracy_questions`

## 7. 页面展示建议

第一版推荐布局：

- 顶部：总览卡片
- 中间：薄弱题目
- 中间：需要关注学生
- 下方：题目正确率表
- 下方：学生完成情况表

这个结构已经足够支撑老师快速判断：

- 班级整体怎么样
- 哪几题最差
- 哪些学生要优先处理

## 8. 一句话结论

前端本次不需要大改架构，只要在现有教师分析页基础上，补消费总览字段、学生状态字段，并把“低正确率学生”显式展示出来，就能把页面从“能看数据”提升到“能支持老师判断学情”。
