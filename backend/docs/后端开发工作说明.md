# 后端开发工作说明

## 1. 后端技术栈

本项目后端采用轻量化 Web API 架构，主要技术包括：

- **FastAPI**：提供 RESTful API 服务，按教师端、学生端、任务、题目、提交等模块拆分路由。
- **SQLite**：作为原型阶段数据库，数据库文件为 `app.db`。
- **SQLAlchemy 2.x**：负责 ORM 数据模型定义、数据库连接、查询与持久化。
- **Pydantic 2.x**：定义请求体、响应体和数据校验结构。
- **Uvicorn**：作为本地开发运行服务器。
- **python-multipart**：支持 CSV 文件上传。
- **CORS 中间件**：允许前端开发环境 `http://localhost:5173` 访问后端接口。
- **标准库 JSON / CSV**：用于题目 JSON 导入、答案 JSON 存储、学生名单 CSV 解析。

项目入口为 `main.py`，其中完成：

- FastAPI 应用创建
- CORS 配置
- 数据表自动创建
- 路由挂载
- SQLite 兼容字段补齐

## 2. 数据模型设计

### 2.1 任务 Task

模型文件：`models/task.py`

主要字段：

- `id`：任务 ID
- `title`：任务标题
- `grade`：年级，例如 `6`、`7`、`8`
- `description`：任务描述
- `status`：任务状态，默认 `draft`
- `created_at`：创建时间
- `questions`：任务下的题目关系

关系：

- 一个 `Task` 对应多道 `Question`
- 学生端根据学生年级筛选可做任务
- 教师端根据任务 ID 查看题目、分析结果和教学摘要

### 2.2 题目 Question

模型文件：`models/question.py`

主要字段：

- `id`：题目主键
- `task_id`：所属任务 ID
- `knowledge_point`：知识点
- `question_type`：题型
- `difficulty`：难度
- `stem`：题干
- `options_json`：选项 JSON 字符串
- `correct_answer_json`：正确答案 JSON 字符串
- `accepted_answers_json`：填空题可接受答案
- `encouragement_json`：鼓励语
- `student_explanation`：给学生看的解析
- `score`：分值

设计说明：

题目结构来自 AI 生成结果，后端以 JSON 字符串形式保存选项、答案、鼓励语等复杂字段，适合当前 SQLite 原型阶段。

### 2.3 学生 Student

模型文件：`models/student.py`

主要字段：

- `id`
- `student_no`：学号，唯一
- `student_name`：姓名
- `grade`：年级
- `class_name`：班级
- `status`：学生状态，默认 `active`
- `created_at`

用途：

- 学生身份校验
- 学生任务筛选
- 教师分析时统计应参与人数
- 按年级、班级过滤学情数据

### 2.4 单题作答记录 AnswerRecord

模型文件：`models/answer_record.py`

主要字段：

- `task_id`
- `question_pk`
- `student_no`
- `student_name`
- `session_id`
- `student_answer_json`
- `correct_answer_json`
- `is_correct`
- `score_awarded`
- `selected_option_explanation`
- `student_explanation`
- `created_at`

约束：

```text
task_id + question_pk + student_no + session_id 唯一
```

作用：

- 保存学生每次练习会话中的单题作答
- 避免同一学生、同一会话、同一题重复写入
- 支撑学生端学习历史和教师端学情统计

### 2.5 整卷提交 Submission / SubmissionDetail

模型文件：

- `models/submission.py`
- `models/submission_detail.py`

当前已实现整份任务提交接口，但当前核心流程更偏向单题实时提交 `AnswerRecord`。`Submission` 和 `SubmissionDetail` 可用于整卷提交场景，仍保留在系统中。

### 2.6 会话记录

系统没有单独建 `Session` 表，而是在 `AnswerRecord.session_id` 中记录一次练习会话。这样可以区分同一学生多次练习同一任务的记录。

## 4. 核心业务逻辑

### 4.1 AI 生成题目如何导入系统

AI 或 Dify 工作流生成题目 JSON 后，前端或调用方将题目数组提交到导入接口。

题目结构由 Pydantic schema 校验，主要包括：

- 知识点
- 题型
- 难度
- 题干
- 选项
- 正确答案
- 填空题可接受答案
- 鼓励语
- 学生解析
- 分值

后端将复杂字段转换为 JSON 字符串保存到数据库。

### 4.2 JSON 题目结构如何解析和保存

在导入题目时，后端将：

```python
options
correct_answer
accepted_answers
encouragement
```

分别通过 `json.dumps(..., ensure_ascii=False)` 保存到 `Question` 表对应字段。

在获取题目、判分和分析时，再通过 `json.loads` 解析回 Python 数据结构。

### 4.3 学生答案如何提交

学生单题提交时需要提供：

- `student_no`
- `session_id`
- `student_answer`

后端校验：

- 学号必须为 8 位数字
- 学生必须存在且状态为 `active`
- 任务必须存在
- 题目必须属于当前任务
- 同一会话同一题不能重复保存

### 4.4 系统如何自动判分

判分逻辑在 `services/scoring.py`。

支持题型：

- `judgment`
- `single_choice`
- `multiple_choice`
- `blank`

判分规则：

- 判断题、单选题：学生答案数组必须与正确答案数组一致。
- 多选题：对学生答案和正确答案排序后比较，避免选项顺序影响判分。
- 填空题：学生答案只要命中 `accepted_answers` 中任一答案即判对。
- 判对得该题分值，判错得 0 分。

同时返回：

- 学生答案
- 正确答案
- 是否正确
- 得分
- 所选选项解释
- 学生解析

### 4.5 如何记录第一次作答

系统在 `AnswerRecord` 中保存作答记录，并设置唯一约束：

```text
task_id + question_pk + student_no + session_id
```

这保证同一学生在同一练习会话中，同一道题只保存一次。

在教师端分析中，系统进一步按：

```text
student_no + task_id + question_pk
```

选取最早的一条记录，作为该学生对该题的第一次作答。

### 4.6 为什么教师分析重点统计第一次作答

课末 5 分钟即时评价关注的是学生在学习后第一时间的真实掌握情况。如果统计反复修改后的结果，可能掩盖学生最初的知识漏洞。

因此教师端分析采用第一次作答，有助于：

- 更真实地反映课堂即时掌握情况
- 找出共性误区
- 判断哪些知识点需要当堂补救
- 避免重复练习对统计结果造成干扰

### 4.7 如何生成学情数据

后端在 `services/analysis.py` 中完成学情分析：

- 按任务查询题目
- 按年级、班级查询 active 学生
- 查询这些学生在当前任务下的作答记录
- 对每个学生、每道题取最早作答
- 计算题目正确率
- 计算学生正确率
- 计算完成进度
- 标记学生状态：`not_started`、`in_progress`、`completed`
- 统计低正确率学生
- 找出正确率最低题目
- 统计各选项被选择次数

完成率计算：

```text
completed_students / assigned_students
```

平均正确率计算：

```text
总正确作答数 / 总作答数
```

题目正确率计算：

```text
该题正确人数 / 该题作答人数
```

薄弱题目规则：

- 分析接口返回正确率最低题目
- 教学摘要接口优先提取正确率低于 `0.6` 的题目

## 5. AI 辅助开发证据

结合代码实现，以下后端工作适合体现“借助生成式人工智能辅助开发”的过程：

### 5.1 数据表结构设计

本项目涉及任务、题目、学生、作答记录、提交详情等多个实体。借助生成式 AI 可以辅助梳理：

- 哪些字段应独立成表
- 哪些字段适合保存为 JSON
- 学生、任务、题目、作答记录之间的关系
- 如何设计唯一约束避免重复提交

### 5.2 API 接口设计

教师端、学生端、任务端接口按不同角色拆分。AI 可以辅助设计：

- 路由命名
- 请求和响应结构
- 错误码和错误信息
- 教师端与学生端接口边界

### 5.3 判分逻辑设计

不同题型判分规则不同。AI 可以辅助比较：

- 单选题与判断题的精确匹配
- 多选题忽略顺序比较
- 填空题多可接受答案匹配
- 返回解析和得分的统一结构

### 5.4 JSON 导入格式设计

Dify 工作流生成的题目需要结构化导入。AI 可辅助形成统一 JSON schema，包括题干、选项、答案、解析、鼓励语等字段，减少前后端约定成本。

### 5.5 学情统计逻辑设计

学情分析涉及完成率、正确率、薄弱题、学生进度、选项误选统计等多维计算。AI 可辅助拆分统计步骤，生成初始代码，再通过测试和调试修正。

### 5.6 错误排查与代码重构

项目中存在从基础分析到详细题目统计、教学摘要、CSV 导入、时间转换等多次迭代。AI 可用于辅助：

- 发现接口字段不足
- 抽取公共逻辑
- 生成文档
- 优化返回结构
- 辅助排查编码、时区、重复提交等问题

说明：当前代码本身不记录“AI 参与开发日志”，以上是结合实现内容可说明的 AI 辅助开发环节。

## 6. 教学价值

### 6.1 支撑课末 5 分钟快速练习

后端支持教师创建任务、导入 AI 生成题目，学生通过学号快速进入对应年级任务，适合课末即时巩固练习。

### 6.2 自动保存学生作答数据

学生单题提交后，系统自动保存答案、正误、得分和解析，避免纸笔练习后教师手工收集数据。

### 6.3 减轻教师批改与统计负担

系统自动判分并统计：

- 学生是否完成
- 每题正确率
- 班级平均正确率
- 低正确率学生
- 高频错误选项

教师无需逐题人工统计。

### 6.4 帮助教师发现薄弱知识点

题目绑定知识点，分析接口和教学摘要接口可以汇总低正确率题目和薄弱知识点，帮助教师判断哪些内容需要复讲。

### 6.5 支持精准教学反馈

选项统计能够反映学生常见误选。例如某个错误选项被大量选择，说明学生可能存在共同误区。教师可据此进行针对性讲解。

### 6.6 为后续教学调整提供依据

教学摘要接口将统计数据整理成结构化 JSON 和可复制文本，便于教师保存、分享或输入大模型生成教学建议，为后续教学调整提供数据基础。

## 后端开发工作摘要（300字以内）

本项目后端基于 FastAPI、SQLite、SQLAlchemy 和 Pydantic 构建，服务于初中信息科技课堂课末 5 分钟即时评价场景。系统支持教师创建练习任务、导入 Dify 工作流生成的结构化题目 JSON、导入学生名单，学生可按学号查询对应年级任务并提交单题答案。后端根据题型自动判分，保存学生作答记录，并以首次作答为核心依据生成学情分析，包括完成率、平均正确率、题目正确率、选项误选统计、学生进度和薄弱题目。系统还提供教学摘要接口，将班级整体情况、薄弱知识点、薄弱题目和学生分层整理为结构化 JSON 与可复制文本，帮助教师快速发现共性问题，减轻批改统计负担，并为后续教学调整和大模型生成教学建议提供数据基础。
